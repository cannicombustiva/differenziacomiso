# Admin writes authorized at the DB layer; subscriptions are service-role-only

Write access to `waste_types`, `collection_schedule`, `riciclabolario`, and `announcements` is authorized in Postgres RLS by **`admins`-table membership**, not by mere authentication. Each write policy tests `EXISTS (SELECT 1 FROM admins WHERE email = lower(auth.jwt()->>'email'))`, wrapped in a `SECURITY DEFINER` `is_admin()` helper to keep policies readable and avoid RLS recursion on the `admins` lookup. The Next.js layer (`middleware.ts`, `requireAdmin()`) performs the same check, so the boundary is enforced in both places (see [CONTEXT.md](../../CONTEXT.md): Admin).

We do this because the anon key ships to every browser and Supabase email signup is on by default — a `TO authenticated USING (true)` policy would let anyone who registers an account write directly to the database, bypassing the application entirely. Gating on `admins` membership makes "authenticated" insufficient; the DB enforces the real boundary regardless of signup configuration. (Public signups are also disabled as belt-and-braces.)

`push_subscriptions` holds push credentials (`endpoint`, `keys_p256dh`, `keys_auth`) and is reached **only** through service-role server routes (`/api/push/*`, the daily cron) — no client ever queries it directly. It therefore carries **no permissive RLS policies**: RLS is enabled (deny-by-default for anon and authenticated) and only the service role, which bypasses RLS, can read, insert, or delete. This replaces an earlier set of policies in which a `FOR ALL USING (true)` policy with no `TO` clause defaulted to `public`, exposing every subscriber's push credentials to anonymous read and delete.
